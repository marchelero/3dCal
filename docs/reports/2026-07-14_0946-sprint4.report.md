## Nits (no bloquean)

1. **Lazy ListView ŌåÆ SingleChildScrollView**: la calculator page cambio de
   `ListView` a `SingleChildScrollView + Column`. Esto es eager build (todos
   los widgets siempre en el tree), que es lo que se necesita para el smoke
   test. Para listas largas seria un problema; aqui OK porque la pagina
   tiene <20 widgets.
2. **Cambio de orden en el form**: el orden de los `DecimalInputField` cambio
   (Peso, Precio, Gramos, Tiempo, ...) al refactor 4.8. El smoke test viejo
   usaba `fields.at(N)` (orden posicional), tuve que migrar a
   `widgetWithText(DecimalInputField, 'Label')`. Vale para tests futuros.
3. **Tests reducidos desde Sprint 3**: 8+5 widget tests para filaments/
   printers, 2 unit tests para printers_notifier. PrintersPage/Form sin
   tests (espejo trivial de Filaments). User pidio tests minimos.
4. **`activePrinterIdProvider` no persiste**: vive solo en memoria de Riverpod.
   Si se reinicia la app vuelve al default. Sprint futuro lo migra a
   `SettingsRepository` si se quiere.
5. **AnimatedList insert/remove via post-frame callback**: el notifier muta
   state primero, el page sincroniza controllers, y `addPostFrameCallback`
   dispara el `insertItem`/`removeItem` en el siguiente frame. Esto es
   funcional pero fragil ŌĆö si Riverpod cambia el orden de las operaciones
   podria desincronizar. Considerar en refactor: usar `ref.listen` para
   reaccionar al state.

## Decisiones tecnicas

- **`AsyncNotifier.update` colision**: el base class de Riverpod tiene
  `update()`. Renombrado a `updateFilament` / `updatePrinter` para evitar
  override accidental.
- **Type inference pitfall**: `state.valueOrNull?.firstWhere((f) => f.id == ...)`
  infiere `f` como `Object?`. Workaround: extraer a local `final list = ...`
  y luego `list.firstWhere(...)`.
- **Filament.pricePerBobbin / gramsPerBobbin** son `double` (drift `RealColumn`),
  no `Decimal`. En `setAsDefault` se reconvierte via
  `Decimal.parse(current.pricePerBobbin.toString())`.
- **Eager build con SingleChildScrollView**: el smoke test necesitaba que
  `find.text` encontrara widgets aunque estuvieran fuera del viewport de 600px
  del test env. `ListView` no los construye lazy. Trade-off: siempre se
  construye todo el form, no se ahorra memoria.
